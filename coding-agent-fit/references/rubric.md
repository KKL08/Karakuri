# Coding Agent 接入评测标准 (Rubric v4)

这套标准评测云服务、API 平台或开发者工具对 Coding Agent 的支持程度。核心目标是：开发者把接入任务交给 Claude Code、Cursor、Codex、Trae 等 Coding Agent 之后，Agent 能否找到服务入口、读懂资料、写出代码，并真正调用跑起来。

评测前先确认三件事：

1. 目标网站提供什么服务。
2. 开发者通常希望 Coding Agent 完成什么接入任务。
3. Agent 从发现入口到跑通第一个调用，中间可能卡在哪些步骤。

## 站点类型

先判断目标属于哪类。混合型站点可以同时标注多个类型，但评分时以主要接入任务为准。

| 类型 | 说明 |
|------|------|
| 云服务 / API 平台 | 提供在线服务、REST/GraphQL API、SDK、控制台、密钥或 OAuth 接入。 |
| 开发者工具 | 主要提供 CLI、SDK、框架、构建工具或本地工具链。 |
| Agent 工具 | 主要提供 MCP、Skill、Agent 配置、工具服务或流程模板。 |
| 文档型站点 | 主要提供知识库、教程、参考文档，没有明确 API 或工具接入。 |
| 交易型服务 | 接入路径包含支付、下单、授权购买、订阅或 agentic commerce。 |

## 维度与权重

| # | 维度 | 权重 |
|---|------|------|
| 1 | 服务入口与发现 | 15% |
| 2 | 接入文档质量 | 30% |
| 3 | Agent 辅助工具 | 20% |
| 4 | 接入阻碍与风险 | 20% |
| 5 | 维护与反馈机制 | 15% |

每个维度先按证据项记分，再映射为 1-5 分。总分计算方式：

```text
维度贡献 = (维度得分 - 1) / 4 * 维度权重
总分 = 所有维度贡献相加
```

## 等级

| 等级 | 分数区间 | 含义 |
|------|----------|------|
| S | 90-100 | Agent 基本可以自己完成接入 |
| A | 75-89 | 整体顺畅，少量地方需要人工判断 |
| B | 60-74 | 可以接入，但会遇到几个明显卡点 |
| C | 40-59 | 勉强可接，人工介入较多 |
| D | 20-39 | 交给 Agent 接入风险高 |
| F | 0-19 | 缺少基本接入条件 |

最终报告除总分外，还要给出两个辅助判断：

- **Agent 接入把握**：高 / 中 / 低。
- **人工介入点**：无需 / 需要拿 key / 需要补 API 判断 / 需要人工排障 / 需要联系销售或支持。

---

## 1. 服务入口与发现 (15%)

评估 Coding Agent 是否能快速找到正确的开发者入口，避开官网、营销页、博客和旧文档带来的干扰。

### 证据项

| 证据 | 分值 |
|------|------|
| 官网导航、开发者入口或文档首页能直接进入 docs / API / SDK / developer portal | 3 |
| 有 `llms.txt`，位于根路径或文档挂载路径，并指向关键文档 | 3 |
| 有 `llms-full.txt` 或分层 `llms.txt` 索引 | 2 |
| sitemap、robots、Link header、HTML link 中至少一种能帮助发现文档 | 2 |
| 搜索结果和页面标题能区分 docs、API reference、SDK、pricing、support | 2 |
| 入口能覆盖 quickstart、API reference、SDK、auth、examples 等主要接入路径 | 3 |

### 分数锚点

| 证据分 | 维度分 | 判断 |
|--------|--------|------|
| 13-15 | 5 | 入口清楚，机器和人都容易找到关键文档。 |
| 10-12 | 4 | 主要入口清楚，少量 AI 发现信号缺失。 |
| 7-9 | 3 | 人能找到，Agent 需要多轮搜索或推断。 |
| 4-6 | 2 | 入口分散，Agent 容易走错。 |
| 0-3 | 1 | 找不到稳定开发者入口。 |

注意：根路径 `/llms.txt` 404 不能直接判缺失。必须检查输入 URL 推导出的文档挂载路径，如 `/docs/llms.txt`、`/developer/llms.txt`、`/developer/docs/llms.txt`。

---

## 2. 接入文档质量 (30%)

评估 Agent 找到文档后，能否读懂资料并完成接入。这里同时看机器可读性和接入资料完整性。

### 证据项

| 证据 | 分值 |
|------|------|
| 有 quickstart，覆盖创建 key、安装依赖和第一个成功调用 | 4 |
| 有完整 API reference 或等价接口说明 | 4 |
| 有机器可读 API spec，如 OpenAPI、Swagger、API Catalog、schema | 4 |
| 鉴权说明清楚，包括 API key、OAuth、token scope、环境变量或权限模型 | 3 |
| 请求和响应 schema 清楚，有字段含义、必填项和示例 | 3 |
| 有 SDK 文档或主流语言示例，示例可以直接复制 | 3 |
| 错误码、HTTP 状态码、限流、重试、超时、幂等等生产细节清楚 | 3 |
| 版本、变更、弃用、兼容性说明清楚 | 2 |
| 文档支持 Markdown、`.md`、`Accept: text/markdown`，或页面结构低噪声 | 2 |
| 文档组织适合 Agent 定位，不只有营销页或动态 UI | 2 |

### 分数锚点

| 证据分 | 维度分 | 判断 |
|--------|--------|------|
| 26-30 | 5 | Agent 基本能独立生成可运行接入代码。 |
| 21-25 | 4 | 接入资料完整，少数生产细节要补查。 |
| 15-20 | 3 | 能跑通基本调用，但需要人工补判断。 |
| 8-14 | 2 | 资料零散，Agent 容易写错或漏关键约束。 |
| 0-7 | 1 | 主要接口、鉴权或调用方式不可判断。 |

硬规则：

- 如果目标是云服务或 API 平台，但缺少 API reference 或鉴权说明，本维度最高 3 分。
- 如果 API reference 和鉴权说明都缺失，且没有等价 SDK/CLI 文档，本维度最高 2 分。
- CLI、MCP、Skill 可以帮助接入。API reference、鉴权、错误码、限流和版本说明仍要单独检查。

---

## 3. Agent 辅助工具 (20%)

评估服务方提供了哪些可被 Coding Agent 直接使用的工具。CLI、MCP、Skill、rules、配置模板都算，重点看它们能否减少查资料、配环境和反复试错。

### 证据项

| 证据 | 分值 |
|------|------|
| 有 CLI，且安装、登录、配置、调用、状态检查、命令参考清楚 | 4 |
| CLI 适合 Agent 调用，如非交互模式、JSON/结构化输出、稳定退出码、doctor/verify 命令 | 3 |
| 有 MCP server 或等价 agent tool，包含 transport、鉴权、工具 schema、验证方式 | 4 |
| 有 Claude Code、Cursor、Codex、Trae、VS Code 等配置示例 | 3 |
| 有 Skill、Agent rules、AGENTS.md、prompt pack 或流程模板 | 3 |
| 工具能读取 docs/spec、列接口、生成配置、跑示例或诊断错误 | 2 |
| 有官方仓库、包管理器安装方式或版本化发布记录 | 1 |

### 分数锚点

| 证据分 | 维度分 | 判断 |
|--------|--------|------|
| 17-20 | 5 | 提供多种辅助工具，Agent 能直接配置、验证并调用。 |
| 13-16 | 4 | 至少一种工具成熟可用，另有配置或模板支持。 |
| 8-12 | 3 | 有 CLI、MCP、Skill 或配置示例之一，但需要人工补配置或验证。 |
| 4-7 | 2 | 只有零散工具说明，Agent 难以直接使用。 |
| 0-3 | 1 | 没有面向 Agent 或命令行的辅助工具。 |

硬规则：

- 只有 MCP URL，但没有配置、鉴权、工具列表或验证方式，本维度最高 3 分。
- 只有 CLI 名称，但没有安装、登录或调用示例，本维度最高 3 分。
- 只有 prompt 文案，没有真实工具、配置或示例，本维度最高 3 分。
- 没有 CLI/MCP/Skill 不可标为 N/A，因为这正是本维度要评的内容。

---

## 4. 接入阻碍与风险 (20%)

评估 Agent 真实接入时会遇到哪些额外阻力。这个维度反向记分，先记录扣分项，再换算成维度分。

### 扣分项

| 阻碍 | 扣分 |
|------|------|
| 文档或关键 API 需要登录才能看，且没有公开替代说明 | 4 |
| API key、OAuth app 或权限申请流程不清楚，或强依赖人工审批 | 3 |
| 示例代码过时、不可运行，或依赖隐藏环境 | 3 |
| 错误信息不可诊断，失败时无法判断是权限、参数、限流还是服务问题 | 3 |
| 地区、套餐、额度、计费、企业限制不清楚 | 2 |
| 文档高度依赖 JS 动态渲染，静态读取困难，Markdown 不可用 | 2 |
| 反爬、bot block、robots 或内容策略会阻碍正常 Agent 获取文档 | 2 |
| SDK、CLI、MCP 版本与文档不一致 | 2 |
| 安全边界不清楚，如 token scope、数据权限、生产风险、tool safety 没讲 | 2 |
| 社区问题显示常见接入坑，但官方文档没有吸收 | 1 |

### 分数锚点

| 扣分 | 维度分 | 判断 |
|------|--------|------|
| 0-2 | 5 | 基本无明显阻碍。 |
| 3-5 | 4 | 有小摩擦，但不影响主路径。 |
| 6-9 | 3 | Agent 会遇到卡点，需要人工介入。 |
| 10-14 | 2 | 接入风险高，容易反复试错。 |
| 15+ | 1 | Agent 很难稳定完成接入。 |

报告必须写出 Agent 最可能卡住的一步。没有这个判断，分数不算完整。

---

## 5. 维护与反馈机制 (15%)

评估服务长期接入是否可靠。Agent 接入跑通一次还不够，还要能应对版本变化、服务故障和文档缺口。

### 证据项

| 证据 | 分值 |
|------|------|
| 有 changelog / release notes，能定位 API、SDK、CLI、MCP 变化 | 3 |
| 有 status page 或服务可用性说明 | 2 |
| 有版本迁移、弃用策略或兼容性说明 | 2 |
| 有文档反馈入口，能定位到页面或具体问题 | 2 |
| 有 support、工单、邮箱、社区、GitHub issues 等求助渠道 | 2 |
| 有 SDK、CLI、MCP 的版本发布记录或官方仓库 | 2 |
| 有安全、权限、数据处理、合规或企业支持说明 | 2 |

### 分数锚点

| 证据分 | 维度分 | 判断 |
|--------|--------|------|
| 13-15 | 5 | 维护信号完整，变更和求助路径清楚。 |
| 10-12 | 4 | 主要维护机制齐全，少数渠道不清楚。 |
| 7-9 | 3 | 有更新和求助渠道，但分散或不够具体。 |
| 4-6 | 2 | 只有基础联系方式或零散 changelog。 |
| 0-3 | 1 | 看不出维护状态和反馈路径。 |

## N/A 规则

尽量少用 N/A，避免分数被洗白。只有这些情况可标 N/A：

- 非 API、非云服务、非开发者工具，才可让部分 API spec 证据项 N/A。
- 不涉及交易时，不检查 commerce 相关标准。
- 没有 OpenAPI 不自动 N/A。API 平台缺 OpenAPI 是缺口。
- 没有 CLI、MCP 或 Skill 不可 N/A。它们属于 Agent 辅助工具维度的评测对象。

如果某个证据项确实 N/A，该维度内剩余证据项按比例归一化，并在报告里说明原因。
