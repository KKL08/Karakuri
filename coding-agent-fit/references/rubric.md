# Coding Agent 接入评测标准 (Rubric v5)

评测云服务、API 平台或开发者工具对 Coding Agent 的支持程度：开发者把接入任务交给 Claude Code、Cursor、Codex、Trae 等 Coding Agent 之后，Agent 能否找到入口、读懂资料、写出代码、真正跑通，并且长期可靠。

评测前先确认三件事：

1. 目标网站提供什么服务。
2. 用户的接入需求是什么（默认按站点典型任务推一个，并和用户确认）。
3. 从发现入口到跑通第一个调用，Agent 中间可能卡在哪。

## 站点类型

| 类型 | 说明 |
|------|------|
| 云服务 / API 平台 | 在线服务、REST/GraphQL API、SDK、控制台、密钥或 OAuth 接入 |
| 开发者工具 | 主要提供 CLI、SDK、框架、构建工具或本地工具链 |
| Agent 工具 | 主要提供 MCP、Skill、Agent 配置、工具服务或流程模板 |
| 文档型站点 | 知识库、教程、参考文档，没有明确 API 或工具接入 |

混合型站点按主要接入任务对应的类型取一套；任务 50/50 时取两套加权平均，报告里写明用了哪套。

## 维度与权重

| 维度 | 云服务 / API 平台 | 开发者工具 | Agent 工具 | 文档型站点 |
|------|----|----|----|----|
| 1. 服务入口与发现 | 10% | 10% | 10% | 15% |
| 2. 接入文档质量 | 30% | 30% | 25% | 50% |
| 3. Agent 辅助工具 | 25% | 30% | 35% | 5% |
| 4. 接入阻碍与风险 | 25% | 20% | 20% | 20% |
| 5. 维护与反馈机制 | 10% | 10% | 10% | 10% |

文档是基础（没文档工具救不回来），工具是顶级门槛（不投工具进不了顶级圈），所以文档和工具都给高权重，文档略高。Agent 工具型站点反过来——工具就是产品本身、文档是说明书，工具优先。

## 计分方式

每个维度算三个值：

- **证据比 R**
  - 正向维度（1/2/3/5）：`R = 实际证据点 / 证据上限`
  - 反向维度（4）：`R = max(0, 1 - friction / friction_max)`，friction 按分组互斥后求和
- **定性档 1-5**：由 R 按各维度锚点表映射。用于触发硬规则、便于阅读。
- **置信度**：高 / 中 / 低，机械算出，不进总分公式。

```
维度贡献   = R * 维度权重 * 100
总分      = Σ 维度贡献
总置信度   = worst-of(各维度置信度)
```

**硬规则应用顺序**（先算后取 min）：

1. 算原始贡献 `R * 权重 * 100`
2. 触发硬规则时，贡献 `= min(原始贡献, (硬上限档 - 1) / 4 * 权重 * 100)`
3. 报告里写明触发了哪条规则、原始贡献是多少

## 置信度评级

置信度独立每维度算，不影响维度贡献，但进入报告并影响 "Agent 接入把握" 的定性结论和改进建议排序。

**证据来源等级**

- **T1（强）**：一手机读 — llms.txt / OpenAPI / .md / `Accept: text/markdown` 已读取
- **T2（中）**：HTML 页面已读取，引用具体段落
- **T3（弱）**：仅 page_signals 关键词命中、搜索标题、第三方仓库

**定义**

```
strong_ratio    = T1 命中证据项 / (总证据项 - N/A)
covered_ratio   = (T1 + T2 命中) / (总证据项 - N/A)
dry_run_verified = 本维度是否被阶段 4.5 实跑覆盖
```

**判定**

```
高    若 strong_ratio ≥ 0.5  且  (dry_run_verified 或 covered_ratio ≥ 0.8)
低    若 covered_ratio < 0.4  或  N/A 比例 > 0.4
中    其他
```

置信度评级在报告各维度详情段必须给出 `strong_ratio` / `covered_ratio` / `dry_run_verified` 三个具体值，不能只写"高/中/低"。

## 等级

| 等级 | 分数区间 | 含义 |
|------|----------|------|
| S | 90-100 | Agent 基本可以自己完成接入 |
| A | 75-89 | 整体顺畅，少量需要人工判断 |
| B | 60-74 | 可以接入，遇到几个明显卡点 |
| C | 40-59 | 勉强可接，人工介入较多 |
| D | 20-39 | 交给 Agent 接入风险高 |
| F | 0-19 | 缺少基本接入条件 |

最终报告除总分外，还要给出：

- **Agent 接入把握**：高 / 中 / 低（必须引用阶段 4.5 实跑结果，不能只凭静态分）
- **评测置信度**：高 / 中 / 低（= 总置信度）
- **人工介入点**：无需 / 需要拿 key / 需要补 API 判断 / 需要人工排障 / 需要联系销售或支持

## 1. 服务入口与发现

Coding Agent 是否能快速找到正确的开发者入口，避开官网、营销页、博客和旧文档干扰。

### 证据项（上限 15）

| 证据 | 分值 |
|------|------|
| 官网导航 / 开发者入口 / 文档首页能直接进入 docs / API / SDK / developer portal | 3 |
| 有 `llms.txt`，位于根路径或文档挂载路径，并指向关键文档 | 3 |
| 有 `llms-full.txt` 或分层 `llms.txt` 索引 | 2 |
| sitemap、robots、Link header、HTML link 中至少一种能帮助发现文档 | 2 |
| 搜索结果与页面标题能区分 docs、API reference、SDK、pricing、support | 2 |
| 入口能覆盖 quickstart、API reference、SDK、auth、examples 等主要接入路径 | 3 |

### 分数锚点

| 证据分 | R | 维度档 | 判断 |
|--------|------|--------|------|
| 13-15 | 0.87-1.00 | 5 | 入口清楚，机器和人都容易找到关键文档 |
| 10-12 | 0.67-0.86 | 4 | 主要入口清楚，少量 AI 发现信号缺失 |
| 7-9   | 0.47-0.66 | 3 | 人能找到，Agent 需要多轮搜索或推断 |
| 4-6   | 0.27-0.46 | 2 | 入口分散，Agent 容易走错 |
| 0-3   | 0.00-0.26 | 1 | 找不到稳定开发者入口 |

**注意**：根路径 `/llms.txt` 404 不能直接判缺失。必须检查输入 URL 推导出的文档挂载路径（`/docs/llms.txt`、`/developer/llms.txt`、`/developer/docs/llms.txt`），以及子域（`docs.host` / `developer.host` / `api.host`）。

## 2. 接入文档质量

Agent 找到文档后能否读懂资料并完成接入。同时看机器可读性和接入资料完整性。

### 证据项（上限 32）

| 证据 | 分值 |
|------|------|
| quickstart 覆盖创建 key、安装依赖、第一个成功调用 | 4 |
| 完整 API reference 或等价接口说明 | 4 |
| 机器可读 API spec（OpenAPI / Swagger / API Catalog / GraphQL schema） | 4 |
| 鉴权说明清楚（API key / OAuth / token scope / 环境变量 / 权限模型） | 3 |
| 请求和响应 schema 清楚（字段含义、必填项、示例） | 3 |
| SDK 文档或主流语言示例，可直接复制 | 3 |
| 错误码 / HTTP 状态码 / 限流 / 重试 / 超时 / 幂等等生产细节 | 3 |
| 版本、变更、弃用、兼容性说明 | 2 |
| 支持 Markdown / `.md` / `Accept: text/markdown`，页面结构低噪声 | 2 |
| 文档组织适合 Agent 定位，不只有营销页或动态 UI | 2 |
| **有 AI Agent / LLM 专属章节**（如 llms.txt 顶部 best practices、Agent integration guide） | 2 |

### 分数锚点

| 证据分 | R | 维度档 | 判断 |
|--------|------|--------|------|
| 28-32 | 0.88-1.00 | 5 | Agent 基本能独立生成可运行接入代码 |
| 22-27 | 0.69-0.84 | 4 | 接入资料完整，少数生产细节要补查 |
| 16-21 | 0.50-0.66 | 3 | 能跑通基本调用，但需要人工补判断 |
| 9-15  | 0.28-0.47 | 2 | 资料零散，Agent 容易写错或漏关键约束 |
| 0-8   | 0.00-0.25 | 1 | 主要接口、鉴权或调用方式不可判断 |

### 硬规则

- 云服务 / API 平台缺 API reference 或鉴权说明 → 本维度档 ≤ 3
- API reference 和鉴权说明都缺，且无等价 SDK / CLI 文档 → 本维度档 ≤ 2
- CLI / MCP / Skill 不能替代 API reference、鉴权、错误码、限流、版本说明

## 3. Agent 辅助工具

服务方提供了哪些可被 Coding Agent 直接使用的工具：CLI、MCP、Skill、rules、配置模板。重点看它们能否减少查资料、配环境、反复试错。

### 证据项（上限 20）

| 证据 | 分值 |
|------|------|
| CLI 完整：安装、登录、配置、调用、状态检查、命令参考清楚 | 4 |
| CLI 适合 Agent 调用：非交互模式、JSON / 结构化输出、稳定退出码、doctor / verify 命令 | 3 |
| MCP server 或等价 agent tool：transport、鉴权、工具 schema、验证方式 | 4 |
| Claude Code / Cursor / Codex / Trae / VS Code 配置示例 | 3 |
| Skill、Agent rules、AGENTS.md、prompt pack 或流程模板 | 3 |
| 工具能读取 docs/spec、列接口、生成配置、跑示例或诊断错误 | 2 |
| 官方仓库、包管理器安装方式或版本化发布记录 | 1 |

### 分数锚点

| 证据分 | R | 维度档 | 判断 |
|--------|------|--------|------|
| 17-20 | 0.85-1.00 | 5 | 提供多种辅助工具，Agent 能直接配置、验证、调用 |
| 13-16 | 0.65-0.80 | 4 | 至少一种工具成熟可用，另有配置或模板支持 |
| 8-12  | 0.40-0.60 | 3 | 有 CLI / MCP / Skill / 配置示例之一，需要人工补配置或验证 |
| 4-7   | 0.20-0.35 | 2 | 只有零散工具说明，Agent 难以直接使用 |
| 0-3   | 0.00-0.15 | 1 | 没有面向 Agent 或命令行的辅助工具 |

### 硬规则

- 只有 MCP URL，无配置、鉴权、工具列表或验证方式 → 本维度档 ≤ 3
- 只有 CLI 名称，无安装、登录、调用示例 → 本维度档 ≤ 3
- 只有 prompt 文案，无真实工具、配置或示例 → 本维度档 ≤ 3
- 没有 CLI / MCP / Skill 不可 N/A——这正是本维度要评的对象

### 触发 5 档的多种路径

5 档不要求同时拥有 CLI + MCP + Skill。下面任一条件触发 5 档候选（仍需其他证据补足到 17 分以上）：

- 成熟 CLI（含 doctor / verify + JSON 输出）+ 文档完整
- Remote MCP + local MCP 双轨 + 接入示例
- 官方 Skill / AGENTS.md + 多 Agent runtime 配置示例

## 4. 接入阻碍与风险

Agent 真实接入时会遇到哪些额外阻力。反向记分：先记扣分，再算 friction_ratio。

### 扣分项

扣分项按组互斥（同组内取 max），组间相加。

**组 A — 机器可读性 / 反爬（同根因，取 max）**

| 扣分项 | 分值 |
|--------|-----|
| 文档高度依赖 JS 动态渲染，静态读取困难，Markdown 不可用 | 2 |
| 反爬 / bot block / robots / 内容策略阻碍 Agent 获取文档 | 2 |

组 A 最大扣分 = 2

**组 B — 账户 / 权限墙（同根因，取 max）**

| 扣分项 | 分值 |
|--------|-----|
| 文档或关键 API 需要登录才能看，且无公开替代 | 4 |
| 申请 API access 需要销售 / 企业沟通才能开通 | 4 |
| API key 创建强制 MFA / 2FA / 验证码，Agent 无法替用户完成 | 3 |
| API key、OAuth app 或权限申请流程不清楚 | 0/1.5/3（分级） |

组 B 最大扣分 = 4

**组 C — 文档维护节奏（同根因，取 max）**

| 扣分项 | 分值 |
|--------|-----|
| 示例代码过时、不可运行或依赖隐藏环境 | 0/1.5/3（分级） |
| SDK / CLI / MCP 版本与文档不一致 | 2 |
| 社区问题显示常见接入坑，官方文档未吸收 | 2 |

组 C 最大扣分 = 3

**组 D — 独立卡点（不分组，各扣各的）**

| 扣分项 | 分值 |
|--------|-----|
| 错误信息不可诊断 | 0/1.5/3（分级） |
| API 需要域名 / DNS 验证才能用 | 2 |
| OAuth redirect URI 要求 https，localhost 不收 | 2 |
| SDK 不覆盖主流语言（无 JS / Python / Go） | 2 |
| 免费层 rate limit 低到影响 Agent 开发体验（< 60 req/h 量级） | 1 |
| 地区、套餐、额度、计费、企业限制不清楚 | 2 |
| 安全边界不清楚（token scope / 数据权限 / 生产风险 / tool safety） | 2 |
| TLS 证书无效或链断 | 2 |

组 D 最大扣分 = 16

**friction_max = 2 + 4 + 3 + 16 = 25**

### 分级扣分项

| 项 | 0 分 | 1.5 分 | 3 分 |
|------|------|--------|------|
| 错误诊断 | 状态码 + 字段级原因 | 状态码 + 模糊 message | 200 + 不可读 body |
| 示例可跑 | 复制即跑 | 一两处需手改 | 全跑不通 |
| API key 流程 | < 3 步拿到 | 3-6 步 / 需邮箱验证 | > 6 步 / 需人工审批 |

### friction_ratio 锚点

| friction_ratio | 维度档 | 判断 |
|----------------|-------|------|
| 0.00-0.10 | 5 | 基本无明显阻碍 |
| 0.11-0.25 | 4 | 有小摩擦，不影响主路径 |
| 0.26-0.45 | 3 | Agent 会遇到卡点，需人工介入 |
| 0.46-0.70 | 2 | 接入风险高，反复试错 |
| 0.71+     | 1 | Agent 难以稳定完成接入 |

### "最可能卡住的一步" 挑选规则

报告必须写出 Agent 最可能卡住的一步。按优先级挑：

1. 阶段 4.5 实跑实际卡住的位置（最硬证据）
2. 扣分 ≥ 3 且无明显 workaround 的项
3. 多个候选时，选最早出现在接入流程里的（步骤 1 卡 > 步骤 5 卡）

### 实跑映射要求

阶段 4.5 实跑出现的卡点必须先映射到本维度某个扣分项；映射不上 = 报告里提议新扣分项作为评测产出。

## 5. 维护与反馈机制

服务长期接入是否可靠。Agent 跑通一次不够，还要能应对版本变化、服务故障和文档缺口。

### 证据项（上限 15）

| 证据 | 分值 |
|------|------|
| changelog / release notes，能定位 API、SDK、CLI、MCP 变化 | 3 |
| status page 或服务可用性说明 | 2 |
| 版本迁移、弃用策略或兼容性说明 | 2 |
| 文档反馈入口，能定位到页面或具体问题 | 2 |
| support、工单、邮箱、社区、GitHub issues 等求助渠道 | 2 |
| SDK / CLI / MCP 版本发布记录或官方仓库 | 2 |
| 安全、权限、数据处理、合规或企业支持说明 | 2 |

### 分数锚点

| 证据分 | R | 维度档 | 判断 |
|--------|------|--------|------|
| 13-15 | 0.87-1.00 | 5 | 维护信号完整，变更和求助路径清楚 |
| 10-12 | 0.67-0.86 | 4 | 主要维护机制齐全，少数渠道不清楚 |
| 7-9   | 0.47-0.66 | 3 | 有更新和求助渠道，但分散或不够具体 |
| 4-6   | 0.27-0.46 | 2 | 只有基础联系方式或零散 changelog |
| 0-3   | 0.00-0.26 | 1 | 看不出维护状态和反馈路径 |

## N/A 规则

按站点类型选权重表已覆盖大部分异质性，N/A 仅限两种：

1. 文档型站点：API spec / SDK 相关证据项可 N/A
2. 评测对象实际能力范围之外（如纯前端工具不评 OAuth）

**不准 N/A**：

- API 平台缺 OpenAPI、缺 CLI / MCP / Skill — 都属于该类型 rubric 要评的对象
- "小服务做不到"不构成 N/A 理由，让证据比自然偏低即可

N/A 时，该维度的证据上限按剩余有效证据项重算，报告说明原因。
